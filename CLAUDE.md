# CLAUDE.md

## Project Overview

Cloudflare Emails is a transactional email API built on Cloudflare Workers using Hono, Zod, and the Cloudflare Email Sending binding. It exposes a REST API for sending emails without a third-party provider.

Stack: **Hono** (routing + JSX renderer) · **Zod** (request validation) · **Vite** + `@cloudflare/vite-plugin` (build) · **Wrangler** (deploy/types) · **TypeScript** (strict mode)

---

## Folder Structure

```
src/
  index.tsx          # App entry — mounts middleware and routes
  renderer.tsx       # Hono JSX renderer (HTML shell)
  style.css          # Global styles
  routers/           # One file per resource/domain
    email.ts         # POST /v1/email/send
public/              # Static assets served as-is
wrangler.jsonc       # Cloudflare Workers config (bindings, compat date)
vite.config.ts       # Vite + Cloudflare + SSR plugin config
tsconfig.json        # Strict ESNext, hono/jsx
worker-configuration.d.ts  # Auto-generated — do not edit manually
```

## Naming Conventions

| Thing | Convention | Example |
|---|---|---|
| Files (routers) | `kebab-case.ts` | `email.ts`, `api-key.ts` |
| Files (components) | `kebab-case.tsx` | `renderer.tsx` |
| Exported router instances | camelCase, match filename | `email`, `apiKey` |
| Zod schemas | camelCase + `Schema` suffix | `sendBodySchema` |
| Bindings (wrangler) | SCREAMING_SNAKE_CASE | `EMAIL` |
| Environment interface | `CloudflareBindings` | (generated by `cf-typegen`) |

## Code Rules

- **Every router lives in `src/routers/`** — one file per resource (e.g. `email.ts`, `attachments.ts`). Mount it in `src/index.tsx` with `app.route()`.
- **Validate all external input with Zod** before touching `c.env` or any business logic. Use `safeParse` and return structured errors on failure.
- **No `any` except at error boundaries** — `catch (err: unknown)`, then narrow. The `err.code` pattern in `email.ts` is acceptable only there.
- **Strict TypeScript** — `strict: true` is on; never disable it or use `@ts-ignore` without a comment explaining why.
- **No comments on obvious code** — only comment non-obvious invariants or Cloudflare-specific workarounds.
- **Do not edit `worker-configuration.d.ts` by hand** — regenerate with `npm run cf-typegen` after changing `wrangler.jsonc`.
- **Hono generics** — always pass `{ Bindings: CloudflareBindings }` to `new Hono()` so binding types are inferred correctly.

---

## Adding a New Route

1. Create `src/routers/<resource>.ts`.
2. Define a `new Hono<{ Bindings: CloudflareBindings }>()` instance.
3. Add Zod schema(s) at the top of the file.
4. Implement handlers using `safeParse` for request bodies.
5. Export the router instance as a named export.
6. Mount in `src/index.tsx`: `app.route('/v1/<resource>', <router>)`.

## Wrangler / Bindings

- Email binding is declared in `wrangler.jsonc` under `send_email` with name `EMAIL`.
- After adding a new binding, run `npm run cf-typegen` to update `worker-configuration.d.ts` and keep `CloudflareBindings` in sync.
- Do not hardcode secrets — use Wrangler secrets (`wrangler secret put`) and reference them via `c.env`.

---

## Error Response Shape

All error responses must follow this shape:

```ts
{ error: string, message?: string, issues?: ZodIssue[] }
```

- Validation failures → `400` with `{ error: 'Validation failed', issues: [...] }`
- Runtime/binding errors → `500` with `{ error: err.code ?? 'UNKNOWN_ERROR', message: err.message }`
- Always use `c.json(...)` — never `new Response(...)` directly.
