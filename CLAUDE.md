# CLAUDE.md

## Project Overview

Cloudflare Emails is a transactional email API built on Cloudflare Workers using Hono, Zod, and the Cloudflare Email Sending binding. It exposes a REST API for sending emails without a third-party provider.

Stack: **Hono** (routing + JSX renderer + factory) · **Zod** (request validation) · **Vite** + `@cloudflare/vite-plugin` (build) · **Wrangler** (deploy/types) · **TypeScript** (strict mode)

---

## Folder Structure

```
src/
  index.tsx          # App entry — mounts middleware and routes
  factory.ts         # Hono factory with CloudflareBindings — use for all apps/middleware
  renderer.tsx       # Hono JSX renderer (HTML shell)
  style.css          # Global styles
  routers/           # One file per resource — only routing + handler glue
    email.ts         # POST /v1/email/send
  schemas/           # Zod schemas — one file per resource
    email.ts
  errors/            # Binding/service error maps — one file per resource
    email.ts
public/              # Static assets served as-is
wrangler.jsonc       # Cloudflare Workers config (bindings, compat date)
vite.config.ts       # Vite + Cloudflare + SSR plugin config
tsconfig.json        # Strict ESNext, hono/jsx
worker-configuration.d.ts  # Auto-generated — do not edit manually
```

## Naming Conventions

| Thing | Convention | Example |
|---|---|---|
| Files (routers) | `kebab-case.ts` | `email.ts`, `api-key.ts` |
| Files (components) | `kebab-case.tsx` | `renderer.tsx` |
| Exported router instances | camelCase, match filename | `email`, `apiKey` |
| Zod schemas | camelCase + `Schema` suffix | `sendBodySchema` |
| Error maps | camelCase + `ErrorMap` suffix | `emailErrorMap` |
| Bindings (wrangler) | SCREAMING_SNAKE_CASE | `EMAIL` |
| Environment interface | `CloudflareBindings` | (generated by `cf-typegen`) |

## Code Rules

- **Routers are thin** — `src/routers/` files contain only route registration and handler glue. No schemas, no error maps, no constants.
- **Schemas in `src/schemas/`** — one file per resource, named to match the router. Import into the router; never define inline.
- **Error maps in `src/errors/`** — one file per resource. Maps binding/service error codes to HTTP status + user-facing message. Import into the router; never define inline.
- **Validate all external input with Zod** before touching `c.env` or any business logic. Use `safeParse` and return structured errors on failure.
- **No `any` except at error boundaries** — `catch (err: unknown)`, then narrow with a cast.
- **Strict TypeScript** — `strict: true` is on; never disable it or use `@ts-ignore` without a comment explaining why.
- **No comments on obvious code** — only comment non-obvious invariants or Cloudflare-specific workarounds.
- **Do not edit `worker-configuration.d.ts` by hand** — regenerate with `pnpm run cf-typegen` after changing `wrangler.jsonc`.
- **Factory pattern** — never use `new Hono<{ Bindings: CloudflareBindings }>()` directly. Always use `factory.createApp()` from `src/factory.ts`. This centralizes the `Env` type so routers and middleware get binding inference for free.

## Adding a New Route

1. Create `src/schemas/<resource>.ts` — define and export Zod schemas.
2. Create `src/errors/<resource>.ts` — define and export the error map if the route calls a binding.
3. Create `src/routers/<resource>.ts` — use `factory.createApp()` from `../factory`, import schemas and error map, implement handlers.
4. Export the router instance as a named export.
5. Mount in `src/index.tsx`: `app.route('/v1/<resource>', <router>)`.

## Wrangler / Bindings

- Email binding is declared in `wrangler.jsonc` under `send_email` with name `EMAIL`.
- After adding a new binding, run `pnpm run cf-typegen` to update `worker-configuration.d.ts` and keep `CloudflareBindings` in sync.
- Do not hardcode secrets — use Wrangler secrets (`wrangler secret put`) and reference them via `c.env`.

---

## Error Response Shape

All error responses must follow this shape:

```ts
{ error: string, message?: string, issues?: ZodIssue[] }
```

- Validation failures → `400` with `{ error: 'Validation failed', issues: [...] }`
- Runtime/binding errors → `500` with `{ error: err.code ?? 'UNKNOWN_ERROR', message: err.message }`
- Always use `c.json(...)` — never `new Response(...)` directly.
